### PR #709 post-merge fallout — UNRESOLVED, currently in flight on `4b992a9`

The PR's NEW strict verifier flags any null-hash row as a chain break. Seven new `audit-api.spec.js` tests fail post-merge with `body.integrityVerified === false` after calling backfill. The pre-merge gate didn't run those e2e tests, so the failure surfaced post-merge.

**WIP fix shipped at `4b992a9`** by the fix-agent (stopped mid-flight when you said "go to office"):

- **writeAudit fork detection** — when the latest row for a tenant has a null hash (pre-#558 legacy state), the fail-soft fallback used to silently anchor new rows on `genesisFor(tenantId)`, forking the chain. The fix runs an inline `backfillTenantChain()` first, re-reads the tail, then only falls back to GENESIS if backfill itself throws.
- **backfillTenantChain fork repair** — distinguishes content tampering (recompute under STORED prevHash doesn't match stored hash → 409) from chain re-ordering (content recomputes correctly under stored prevHash, but stored prevHash doesn't match the `[createdAt asc, id asc]` walk → safely re-stamp). Lets backfill actually achieve `integrityVerified: true` against freshly-seeded tenants without losing tamper-evidence.
